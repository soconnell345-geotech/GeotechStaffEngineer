# v5.0 — `deepagents` Migration Plan

> **Status:** ACTIVE plan for the `v5.0-deepagents` branch.
> **Decision (2026-06-05):** owner chose the **strangler-behind-a-flag** approach and
> greenlit dispatching the offline Phase 0–1 build now.
> **Current release:** 4.6.5. **Target:** 5.0.0.
> **Scope:** the *agent layer* only (`funhouse_agent/`). The 36 analysis modules, the
> `geotech-references` submodule, and the validated math are **not** rewritten — they
> are the asset we wrap, not replace.

---

## 1. Executive summary (plain language)

Today the "brains" that lets an LLM drive our 900+ geotechnical calculations is a
**hand-built agent loop** (`funhouse_agent/`). It works and has genuinely smart pieces,
but a meaningful chunk is plumbing we maintain by hand: reading the model's replies,
guessing whether a reply is "a step" or "the final answer," managing conversation
memory, and wiring up our reviewer and reference-lookup helpers one by one.

`deepagents` is a now-mature open-source **agent harness** (LangChain team, currently
**v0.6.8**) that provides that plumbing as a maintained, batteries-included library.
The pieces we hand-rolled — the loop, memory management, sub-agents — are its built-in
features, plus capabilities we don't have today (**planning, a scratch/memory
filesystem, automatic context summarization, durable resumable sessions, live
streaming, and human-approval gates**).

**This is NOT a from-scratch rewrite.** We keep everything valuable — our 52
calculation adapters and the "discovery" tool design — and swap out *only* the brittle
hand-built loop for the `deepagents` engine, building it **side-by-side behind a flag**
so 4.6.5 keeps shipping untouched and we prove the new agent is better on our own
68-question test set before cutover.

**Headline gains:**

| Win | Why it matters |
|-----|----------------|
| Delete the brittle text-parsing loop | Removes the root cause of the known **"fabricated success on tool error"** bug (HANDOFF P1) |
| **Planning** (`write_todos`) | Agent lays out + tracks multi-step jobs (classify → SPT → bearing → settlement); more reliable, more transparent |
| **Cross-session memory** (`StoreBackend`) | Remember a project's soil profile / prior results between sessions |
| **Context summarization + file offload** | Long "run 12 analyses and compare" sessions stop blowing the context window |
| **Cleaner specialist sub-agents** | `consult_references` + reviewer become first-class isolated sub-agents; new ones (report-builder, QC) become trivial |
| **Durable + streamable + human-in-the-loop** | Resume a session, watch the agent reason live, optionally approve a calc before it runs |
| **Less code we own** | Inherit LangChain/LangGraph's maintained loop + LangSmith tracing |

**Costs (honest):** a heavy dependency (`langchain`/`langgraph`) **in the agent layer
only** (core scientific package stays clean — ship it as an extra); re-bridging the
Databricks/Funhouse `PrompterAPI` backend through LangChain (the main risk — §8); a
rewrite of the notebook chat UI; re-pointing eval/triage tooling at the new agent.

### Two design ideas worth preserving
1. **Discovery-based tool model** — 4 meta-tools (`list_agents`→`list_methods`→
   `describe_method`→`call_agent`) over a 900-method registry, instead of 900 tools. A
   good answer to "too many tools." **Keep it.**
2. **Reference access via a scoped sub-agent** — `consult_references` isolates the 21
   reference modules off the primary's surface. Exactly the **context-isolation**
   pattern `SubAgentMiddleware` is built for — a 1:1 port.

## 3. `deepagents` today (v0.6.8, June 2026)

A composable **middleware stack on LangGraph**. `create_deep_agent(model, tools,
system_prompt, subagents, ...)` auto-attaches:

- **`TodoListMiddleware`** — `write_todos` planning tool + prompting.
- **`FilesystemMiddleware`** — `ls`/`read_file`/`write_file`/`edit_file` over
  `StateBackend` (ephemeral), `StoreBackend` (persistent across threads),
  `CompositeBackend` (route `/memories/` → persistent). `read_file` returns
  images/PDFs/audio as multimodal content blocks.
- **`SubAgentMiddleware`** — a `task` tool + custom subagents, each a dict
  `{name, description, system_prompt, tools, model?, middleware?}`. Point = **context
  isolation**. (Advanced: `CompiledSubAgent` for a custom graph; async/remote subagents
  via the Agent Protocol since v0.5.)
- **`SummarizationMiddleware`** — auto-compresses history near token limits.
- Opt-in: `ToolRetryMiddleware`, `Model/ToolCallLimitMiddleware` (runaway guards),
  `HumanInTheLoopMiddleware`, model fallback, PII redaction, Anthropic/OpenAI
  prompt-caching middleware.

From LangGraph: **streaming, checkpointing/persistence (threads), durable resume,
LangSmith tracing**. Model-agnostic (any tool-calling LLM via a LangChain chat model).

## 5. Recommended approach — strangler, behind a flag *(chosen)*

Build `funhouse_agent` v2 **alongside** v1. New code under `funhouse_agent/deep/`
wraps the **same adapters** as LangChain tools and stands up `create_deep_agent`,
selected behind a flag (`GeotechAgent(backend="deep")` / a `build_agent(...)` factory):
- 4.6.5 / `master` keeps shipping with **zero** behavior change;
- A/B the two agents on the **68-question eval** before cutover;
- v1 stays a fallback for one full release after cutover.

Maps onto the working model: **`v5.0-deepagents` = migration; `master` = 4.6.5 fixes.**

## 7. Mapping `consult_references`/`reviewer` → subagents (concrete)

```python
# sketch — funhouse_agent/deep/agent.py
from deepagents import create_deep_agent
from funhouse_agent.deep.tools import CORE_TOOLS, VISION_TOOLS, REFERENCE_TOOLS

agent = create_deep_agent(
    model=model,                       # LangChain chat model (§8)
    tools=CORE_TOOLS + VISION_TOOLS,   # 4 meta-tools + vision, over the SAME adapters
    system_prompt=DOMAIN_PROMPT,       # trimmed system_prompt.py (no ReAct boilerplate)
    subagents=[
        {
            "name": "references",
            "description": "Reference librarian: NAVFAC DM7, FHWA GEC, UFC, micropile, "
                           "FEMA, NOAA. Cites section/table/figure; reads chart values "
                           "off rendered figures.",
            "system_prompt": CONSULTANT_FRAMING,         # from reviewer.py
            "tools": REFERENCE_TOOLS,                    # meta-tools scoped to REFERENCE_MODULES + figure read-off
        },
        {
            "name": "reviewer",
            "description": "Senior QC reviewer: checks a finished analysis against the references.",
            "system_prompt": REVIEWER_SYSTEM_PROMPT,     # from reviewer.py
            "tools": REFERENCE_TOOLS,
        },
    ],
)
```
The primary's `consult_references` tool call becomes a `task(subagent="references", ...)`
delegation; scoping that today lives in `allowed_agents` is reused verbatim by giving
the subagent only the reference-scoped tools.

## 8. The one technical risk — the Funhouse `PrompterAPI` bridge

Everything downstream depends on this. Today `NativeToolEngine` wraps `prompter.client`
(a raw OpenAI client behind the Databricks/Funhouse proxy) and sends **explicit-dict**
messages because the proxy "can carry null fields (function_call, refusal, audio,
annotations) that some servers reject." LangChain must drive that same proxy. Options,
easiest first:
1. **`ChatOpenAI(base_url=..., api_key=..., default_headers=...)`** at the Funhouse
   endpoint — if the proxy is OpenAI-compatible enough, this "just works."
2. **A thin custom `BaseChatModel`** calling `prompter.client` directly (mirrors today's
   `NativeToolEngine`, preserving message hygiene). Most control; ~1 small class.
3. If the official `funhouse` SDK ships a LangChain adapter (it already ships engine
   classes — `USING_SDK_ENGINES`), prefer that.

**Phase 0 must prove one real multi-tool turn end-to-end on Databricks** — needs the
owner's env + API budget (a subagent on this box can't reach Databricks). Offline, a
subagent proves the rest with the mock/`ClaudeEngine` path.

## 10a. Phase 0–1 findings (2026-06-05) — build notes for Phase 2

The offline tool layer + agent assembly are built and green (`funhouse_agent/deep/`).
Concrete things Phase 2 must carry forward:

1. **`call_agent` needs an explicit `extra="allow"` args schema.** A bare `**kwargs`
   signature does NOT survive `StructuredTool.from_function` (LangChain collapses it to
   one `extra` field and drops top-level keys). We declare `_CallAgentArgs(BaseModel)`
   with `ConfigDict(extra="allow")` so the model's flattened method params flow through
   and get auto-nested — same robustness as v1's `dispatch_native_tool`. Keep this.
2. **The deepagents `task` tool + a built-in `general-purpose` subagent are ALWAYS
   present** and can't be removed. So `reference_mode="off"` can only omit our *named*
   `references`/`reviewer` subagents — it can't remove `task`. The v1 `"after_calc"`
   gating (offer reference access only after a calc runs) has **no direct equivalent**;
   it would need custom middleware. Decide in Phase 2 whether that gating still matters.
3. **`model=` (LangChain chat model) and `engine=` (Funhouse vision) are two different
   objects.** Vision tools call `engine.analyze_image(bytes, prompt)` — a `GenAIEngine`
   method, not a `BaseChatModel` method. Phase 2 must either pass a Funhouse engine
   alongside the LangChain model, or write a small adapter exposing `analyze_image` over
   the LangChain multimodal message API. (`read_reference_figure`/`save_file` work
   without a live engine; `analyze_*` return a clean error when `engine=None`.)
4. **Attachments are build-time-closured, not per-`ask()`.** v1 mutates
   `self._attachments` each turn; here they're captured at `build_deep_agent` time. If
   per-invocation attachments are needed, wire them through deepagents state/context
   (or rebuild tools per request).
5. **Vision-tool arg fidelity** (`read_reference_figure(reference, figure_number,
   prompt)`) should be re-checked against `vision_tools` exact arg contract in Phase 2
   (offline tests only assert it errors cleanly without an engine).
6. **Full agent-loop test needs a real tool-calling model.** `GenericFakeChatModel`
   builds the graph but doesn't emit structured `tool_calls`, so it can't drive
   model→tools→model. Use `ChatAnthropic` (API key) or the Databricks-proxied model.
