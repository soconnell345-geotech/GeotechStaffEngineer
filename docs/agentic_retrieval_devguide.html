The big picture

01Two layers: a retrieval engine, and a harness that drives it

The system splits cleanly along a seam worth copying. The geotech-references submodule is a pure-Python library: digitized data plus a retrieval engine, with no LLM and no agent loop inside it. The funhouse_agent harness is the agent: an engine-agnostic loop that turns the library's functions into tools the model can call. The same library is wrapped two ways — as standalone Foundry functions and as Funhouse adapters — so the retrieval logic is written once and reused.

The recipe in six moves

  1. Digitize once, in plain Python. Turn your corpus into committed data: callable lookups, structured JSON, a figure catalog. No LLM in the library.
  2. Index lazily with SQLite FTS5. Build a BM25 full-text index into a temp DB on first call; rebuild only when source files change.
  3. Expose summary-first tools. search returns ranked summaries; get fetches one full section. Cheap to scan, then drill in.
  4. Add a recall lever. A dependency-free synonym expansion fixes vocabulary gaps that stemming can't (and is far cheaper than embeddings).
  5. Wrap as tools, scope with a sub-agent. A reference-only consultant keeps the primary agent's tool surface small and forces citations.
  6. For charts, find-with-text, read-with-pixels. Locate a figure lexically, render the page, and let a vision model read the value off it.

Inside the references submodule

02How the Python comes together: data → engine → wrappers

Each reference (a manual, a code, an FHWA circular) is one package directory. The data lives in three complementary forms, all written by hand from the source PDF — never auto-generated at query time. The private _*.py modules are the engine that turns that data into ranked, drillable results.

Key idea — the three forms are not redundant; they are three access modalities over the same source. A value that was digitized is served exactly by a Python lookup; narrative is served by text search; and anything only available as a chart is served by figure-search + vision read-off. The agent picks the modality.

The retrieval engine

03SQLite FTS5, built lazily, ranked by BM25, widened by synonyms

There is no vector store and no embedding service. Retrieval is lexical: an FTS5 virtual table with porter stemming, ranked by BM25. The DB is a build artifact — created in a temp dir on first call and rebuilt automatically whenever any source JSON is newer than the index. This keeps the repo clean and the index always fresh.

_retrieval_db.py · the index schema (built once, lazily)
def _build_db(db_path):
    cur.execute("""
      CREATE TABLE sections (
        reference, reference_title, chapter, chapter_title,
        section_id, title, summary, body, key_points, applicability,
        equations_json, figures_json, tables_json )""")
    cur.execute("""
      CREATE VIRTUAL TABLE sections_fts USING fts5(
        title, summary, body, key_points, applicability,
        content='sections', content_rowid='rowid',
        tokenize='porter unicode61' )""")        # ← stemming built in
    # walk every <reference>/text/*.json, INSERT one row per section,
    # then mirror the indexed columns into the FTS table.

def _needs_rebuild(db_path):
    return (not db_path.exists()
            or db_path.stat().st_mtime < _newest_source_mtime())  # self-healing

The public functions are deliberately shaped for an LLM: search is summary-only (the noise-reduction lever), and get is the one place the full body is paid for. Notice the strategy switch inside reference_search — the synonym expansion is applied as a re-rank that pins the literal top-1, so a query that was already good keeps its best hit while weaker queries gain recall.

_retrieval_db.py · search = rank + expand; get = drill in
def reference_search(query, reference=None, chapter=None, limit=5):
    # run() = SELECT … FROM sections_fts MATCH ? ORDER BY bm25(sections_fts) LIMIT ?
    strategy = _qe.EXPANSION_STRATEGY                 # env GEOTECH_RETRIEVAL_EXPANSION
    if strategy == "auto":                         # the default
        literal  = run(query)
        combined = _qe.combined_query(query)          # "(query) OR synonym1 OR synonym2 …"
        rows = _qe.merge_hits([literal[0]], run(combined), limit)   # pin top-1, rerank rest
    if not rows and strategy != "off":            # zero-hit recall fallback
        rows = run(_qe.or_fallback(query))            # AND → OR-of-terms
    return [_row_to_summary(r) for r in rows]   # ← summary only, NO body

def reference_get(reference, section_id):              # the drill-in
    return _row_to_full(row)   # body + key_points + equations(implemented_in→python fn) + figures + tables

Why expansion, not embeddings?

Stemming closes morphological gaps (“pressures”→“pressure”) but not synonym gaps (“bored pile” vs “drilled shaft”, “interface friction” vs δ). A curated, dependency-free synonym table fixes that. Measured lift: recall@5 11% → 44% with zero top-1 disturbance under auto. Text embeddings stay deferred; image embeddings were rejected for axis-labeled charts.

Figures: find with text, read with pixels

_figures_db.figure_search ranks the catalog (same FTS5 pattern) and resolve_pdf() returns (pdf_path, page_index). The harness renders that page at 220 DPI and hands the actual image to a vision model. CLIP-style image embeddings were explicitly rejected — they fit line charts poorly.

Safety in reference_query — the raw-SQL escape hatch opens the DB mode=ro, regex-gates to SELECT/WITH only, rejects a forbidden-keyword set (INSERT/UPDATE/DROP/PRAGMA…), blocks multi-statements, and caps the result at 50 rows. Give the model power and a fence at the same time.

Tools the model sees

04One engine, two front doors — Foundry functions & Funhouse adapters

The retrieval engine is wrapped twice so the same logic ships to two runtimes. For Foundry: agents/references_agent.py exposes six @function-decorated tools. For Funhouse: the thin adapters in funhouse_agent/adapters/ bridge flat JSON ⇄ the engine and are what the harness loads. Both call the very same _retrieval_db / _figures_db.

The agentic loop

05How a question becomes a cited answer

The harness is where “retrieval” becomes “agentic retrieval.” The model isn't handed a context blob — it runs a loop, choosing one tool per round, reacting to each result, and deciding when to drill in, switch modality, or stop. Reference access is delegated to a scoped consult sub-agent so the primary agent never carries the 21-reference surface itself.

ROUND 0User asks“Per GEC-10, what α for drilled-shaft side resistance in clay, su=80 kPa?” hits the primary GeotechAgent.ask().
DELEGATEconsult_referencesPrimary calls one tool: the consultant. A reference-scoped GeotechAgent spins up (allowed_agents=REFERENCE_MODULES, reference_mode="off" so no recursion).
SEARCHreference_searchConsultant runs FTS5 + synonym expansion → ranked summaries. Cheap scan picks the right GEC-10 section.
DRILLreference_get / lookupFetches the full section, or calls the digitized α figure function for the exact value at su=80.
CHART?figure→visionIf the value only lives on a chart: figure_search → read_reference_figure renders the page and reads it off.
ANSWERcited resultConsultant returns a cited answer; primary folds it into the final response. Optional reviewer re-checks vs standards.

# In the primary agent's loop, a tool call named consult_references routes here:
def consult_references(engine, question, max_rounds=10):
    sub = GeotechAgent(
        genai_engine   = engine,                 # same model → native tool calling for free
        allowed_agents = REFERENCE_MODULES,      # dm7, gec*, ufc_*, reference_db, figure_db…
        reference_mode = "off",                 # can't recurse into another consult
        review         = False,
    )
    return sub.ask(CONSULTANT_FRAMING + question).answer   # "act as a reference librarian; CITE everything"

Two loop flavors

Native (OpenAI tools param) when the engine advertises native_tool_calling — structured tool_calls. Otherwise ReAct: the model emits <tool_call> XML that the harness parses. Same dispatch underneath.

reference_mode

anytime (default) always offers the consultant and scopes the primary to analysis modules; after_calc offers it only once a calc has run; off leaves the reference modules directly callable (legacy).

Engine-agnostic

Anything satisfying the GenAIEngine protocol (a chat(), optional analyze_image()) drives it: ClaudeEngine, NativeToolEngine, or the Funhouse Prompter bridge. The loop doesn't care which.

Why it's agentic, not just RAG

06Five design patterns that do the heavy lifting

A vector-DB RAG call stuffs top-k chunks into context and hopes. These patterns instead make the right behavior the path of least resistance — the model is nudged toward verifying rather than guessing.

① Summary-first noise reduction

search returns only one-line summaries; the costly body is paid for exactly once, in get, after the model has chosen the section. Context stays clean; the model scans, then commits.

② Starve the shortcut

A figure search hit carries no chart value — only an identity plus a read_value instruction pointing at read_reference_figure. With nothing to guess from, the vision read-off becomes the easiest path, killing the “answer from the caption / from memory” failure (Fig 4-12 Kp: confabulated ≈9.0 → read-off ≈5.3).

③ Scoped consult sub-agent

Reference access is one tool, not 21 modules, on the primary. The consultant is whitelisted to reference modules and framed as a librarian that must cite. Small tool surface = far less method/module-name guessing.

④ Recall lever > precision risk

Synonym expansion as a re-rank that pins the literal top-1: good queries are undisturbed, weak ones gain recall. A cheap, auditable table beats an embedding service for a closed technical vocabulary.

⑤ Smart method resolution

When the model guesses a method name that's really a parameter value (call_agent('bearing_capacity','vesic')), dispatch detects the selector-value, suggests the real call, or routes a curated alias — turning a dead-end error into progress.

⑥ Self-healing, zero-config index

The FTS DB rebuilds when sources change and lives in a temp dir, so there's no migration step and no stale index. Editable install means data edits are live from source.

Build your own Funhouse bot

07Port the pattern to your domain in five files

Nothing here is geotech-specific. To stand up agentic retrieval over your corpus in Funhouse, copy the seam: a pure-Python retrieval library, thin tool wrappers, and a scoped agent loop. Here's the minimal skeleton.

① retrieval_engine.py — the library (no LLM here)
import sqlite3, tempfile, json, pathlib

def _ensure_db():
    db = pathlib.Path(tempfile.gettempdir()) / "mycorpus_fts.sqlite"
    if _needs_rebuild(db):           # compare db mtime to newest source file
        _build_fts5(db)              # CREATE VIRTUAL TABLE … USING fts5(title, summary, body, tokenize='porter')
    return db

def search(query, limit=5):       # SUMMARY-ONLY — the noise-reduction lever
    rows = _run("SELECT id,title,summary FROM docs_fts WHERE docs_fts MATCH ? "
                "ORDER BY bm25(docs_fts) LIMIT ?", [combined_query(query), limit])
    return [dict(r) for r in rows]

def get(doc_id):               # the drill-in — full body paid for once
    return _fetch_full(doc_id)
② tools.py — wrap each engine fn as a Funhouse/Foundry @function with a docstring the model reads
@function
def corpus_search(query: str, limit: int = 5) -> str:
    """Search the corpus. Returns ranked SUMMARIES only — call corpus_get for the body.
       Run this FIRST to scope, then drill in."""
    return json.dumps(retrieval_engine.search(query, limit))

@function
def corpus_get(doc_id: str) -> str:
    """Fetch the full body of one document by id (use after corpus_search)."""
    return json.dumps(retrieval_engine.get(doc_id))
③ agent — a scoped consultant + a loop (engine-agnostic)
agent = GeotechAgent(                       # or your own loop over the GenAIEngine protocol
    genai_engine   = NativeToolEngine(fh_prompter),   # Funhouse Prompter → native tool calling
    allowed_agents = {"corpus_db"},               # scope: only your retrieval tools
    reference_mode = "anytime",
)
print(agent.ask("What does the spec say about X, and cite the section?").answer)

Do

Keep the library LLM-free · make search summary-only · write tool docstrings for the model (say what to call next) · scope the agent's tools · render+vision for anything that's a chart.

Skip (at first)

A vector DB — start lexical (FTS5 + BM25 + a synonym table); it's cheaper, debuggable, and often enough. Add embeddings only when recall plateaus.

Steal

The starve-the-shortcut signpost · the read-only SQL fence · lazy self-healing index · selector-value method resolution · the pin-top-1 re-rank.

Why a sub-agent instead of just more tools? Every tool you add to the primary widens its choice space and its error rate. Delegating retrieval to a scoped consultant keeps the primary's surface tiny, lets the consultant specialize (the “cite everything” framing), and prevents a reference lookup from accidentally calling a compute tool. In live testing this single change cut method- and module-name guessing materially.
