Read this first

It does not know your site, and it cannot exercise engineering judgment.

What this software is — and is not

  • It is a library of deterministic methods, a probabilistic variability engine, digitized references, and an LLM agent that drives them.
  • It is not a stamped design, a professional opinion, or a substitute for site-specific investigation, characterization, and review.
  • Using this software creates no engineer-of-record relationship and no professional engagement with the authors or contributors.

Your responsibilities as a user

  • A qualified, licensed professional engineer familiar with the site must independently review every input, assumption, method selection, and result before it is relied upon. Outputs are candidate calculations to be checked, not conclusions to be adopted.

Agreement on a benchmark does not certify correctness for any other geometry, parameter range, or loading condition.

Chapter 1What this package is

Geotechnical engineering is the practice of building on and in the ground — foundations, retaining walls, slopes, excavations, embankments. Unlike a steel beam with a certified strength, the geotechnical engineer’s material is the earth itself: heterogeneous, layered, partly saturated, and sampled at only a handful of points across an entire site. Because the ground is variable and only partly known, a single number is never the answer.

Understanding a geotechnical problem means understanding how the answer moves as the inputs move. Analysis is repeated calculation across plausible subsurface and loading conditions — running a chained assortment of industry-standard formulas, empirical correlations, and numerical methods, comparing against the design requirements, and varying assumptions until the design is robust across the uncertainty. The engineer’s job is to understand the range and spread of answers, not one point estimate. The true answer is a distribution.

1.1 What a geotechnical engineer actually does

StepActivityReality
CharacterizeDrill borings, push CPT cones, run lab testsA sparse, noisy picture of strength, stiffness, and groundwater — never the whole truth
IdealizeCollapse the data into a layered soil profileDesign values of φ, c, γ, water table — each an estimate with a spread
AnalyzeRun the method (bearing, settlement, pile, slope)A chained assortment of standard formulas and numerical methods
Check & reviseCompare to requirements; vary assumptions; re-runLoop until the design is robust across the uncertainty

Steps 3 and 4 are repeated calculations across plausible conditions. This toolkit packages those methods as clean, machine-callable Python and puts a reasoning agent on top so the engineer can explore more of the problem in the time they have.

1.2 The lineage: every tool amplified the engineer

Slide rules and design charts → spreadsheets and FEM → scripting and Monte Carlo → LLM agents. Every tool in this lineage did the same thing: it let one engineer explore more of the problem. This project is the next step — not a replacement for judgment, but a multiplier for it. Take the methods a staff engineer uses, make every one a clean function that returns structured data, wrap the variability tooling around them, and drive them with an agent that reads the standards back to you.

1.3 The architecture in four layers

One stack, layered so each concern stays independent:

Everything is SI (m, kPa, kN, kN/m, degrees). Every analyze_*() returns a dataclass with .summary() for human/agent reading and .to_dict() for JSON. Analysis modules never import each other — the routing that combines them lives in the agent layer. That discipline is what lets a result be traced, reviewed, and cited.

2.4 Databricks / Funhouse

Install from a cluster-accessible location — /tmp or a Unity Catalog Volume, not /Workspace (the workspace FUSE mount mangles wheel filenames and does not durably store output files):

%pip install "geotech-staff-engineer[deep,full]"
# or a test wheel uploaded to /tmp or a UC Volume:
%pip install "/tmp/geotech_staff_engineer--py3-none-any.whl[deep,full]"
The typing_extensions / restart-Python note (now handled automatically)

Databricks cluster runtimes pre-import an old typing_extensions (<4.13) at kernel startup, which used to break the langgraph imports behind the [deep] agent with TypeError: TypedDict() got an unexpected keyword argument 'extra_items', requiring dbutils.library.restartPython(). As of v5.4 the package repairs this in place: funhouse_agent.runtime_check.ensure_typing_extensions() runs automatically at the top of the first funhouse_agent.deep import, reloading the freshly-installed typing_extensions≥4.13 so the restart is normally no longer needed. If an even older copy is pinned at cluster scope and the in-place reload can’t win, it raises a clear message that falls back to the restart instruction.

Save calc packages, DXF, and plots to /tmp or a UC Volume and copy them out with dbutils.fs.cp; writes to /Workspace can silently leave an 11-byte placeholder. As of recent releases the file tools defend themselves — they verify the written bytes and return a rescue_path in /tmp if the target did not store the content.

2.5 Reference figures: GEOTECH_REFERENCES_DOCS

The reference figure catalogs (captions, page numbers, cross-links) ship inside the wheel, so figure_search works from a clean install. The source PDF pages are not shipped (large + license). To enable the vision read-off — where the agent renders a catalogued chart and reads a value off it — place the reference PDFs in a folder and set GEOTECH_REFERENCES_DOCS to point at it. On Databricks, copy a docs/ folder of PDFs in and set the variable.

Chapter 3Using the agent

The same validated methods are reachable at three altitudes: import a module and get a dataclass back; wrap that method to return the distribution; or describe the problem in natural language and let the agent run the methods and cite the standards. This chapter is about the third altitude — the agent.

3.1 The v1 agent: GeotechAgent

GeotechAgent is engine-agnostic: it drives any backend that satisfies the small GenAIEngine protocol (a chat() method, optionally analyze_image() for vision). Ship it a Claude engine, a Funhouse PrompterAPI, or a native OpenAI-tool-calling engine.

from funhouse_agent import GeotechAgent, ClaudeEngine, NativeToolEngine

# Claude (reads ANTHROPIC_API_KEY from the environment)
agent = GeotechAgent(genai_engine=ClaudeEngine())

# Databricks / Funhouse with native OpenAI tool calling (recommended there)
agent = GeotechAgent(genai_engine=NativeToolEngine(fh_prompter))

result = agent.ask("2 m strip footing, 1.5 m deep, sand phi=30, c=10, water at 5 m. "
                   "Bearing capacity and FOS - and how sensitive is it to phi? Cite the method.")
print(result.answer)        # the cited answer
print(result.tool_calls)    # every method the agent ran
print(result.rounds)        # ReAct iterations

3.2 How tool dispatch works

The agent never has to know a method signature in advance. It discovers the toolbox with four dispatch tools, then calls the one it needs:

ToolWhat it returns
list_agents()Every available module with a one-line brief
list_methods(agent, category)The methods on a module, grouped by category
describe_method(agent, method)Full parameter documentation — names, types, required flags, allowed values
call_agent(agent, method, params)Execute the calculation; returns the result dict or a helpful error

The dispatch layer is forgiving by design: a method name the model guesses is resolved through a curated alias map; a value that belongs to a selector parameter (say 'vesic') is redirected to the right call with a directive error; and a genuinely unknown name gets a “did you mean” suggestion. This cut the method-name guessing that otherwise dogs tool-using agents.

3.3 References through a scoped consultant

The primary agent does not call the 21 reference modules directly. Reference access is routed through a single consult_references tool backed by a reference-scoped sub-agent — a second GeotechAgent restricted to the reference modules. This keeps the primary agent’s tool surface small (which measurably cut module-name guessing) while still giving it the full reference library, figure search, and vision read-off on demand. The behavior is set by reference_mode: 'anytime' (default) offers the consultant always; 'after_calc' only after a calculation has run; 'off' keeps the legacy direct-call behavior.

3.4 Vision: reading drawings and charts

When the engine supports images, the agent gains vision tools: analyze_image and analyze_pdf_page for site plans and report pages, and read_reference_figure, which finds a design chart by meaning (figure_search) and then renders the actual page and reads a value off it — rather than answering a chart value from memory. Attach files before asking:

with open("cross_section.pdf", "rb") as f:
    agent.add_attachment("section", f.read())
result = agent.ask("Extract the slope geometry from 'section' and find the FOS by Bishop.")

# direct geometry extraction (bypasses the ReAct loop)
geo = agent.extract_geometry_from_pdf(pdf_bytes, page=0)

4.1 Shallow foundations

Sizing and checking spread footings and mats: how much load the ground carries, and how far it settles.

4.1.1 bearing_capacity — Bearing capacity

Problems it solves

Sizing a spread footing or mat and confirming the ground can carry it: the ultimate bearing capacity, the allowable pressure at a chosen factor of safety, and the sensitivity of both to width, embedment, groundwater, eccentricity, and a layered profile.

Methods & theories implemented
  • General bearing-capacity equation with Nc, Nq, Nγ by Vesic, Meyerhof, or Hansen (selectable); Vesic is the default and the AASHTO/GEC-6 factor set.
  • Shape, depth, load-inclination, base-tilt, and ground-slope correction factors (Vesic form).
  • Eccentric loading via the Meyerhof effective-area (B′, L′) reduction.
  • Two-layer (strong-over-weak / weak-over-strong) bearing with load spread.
  • Groundwater in the failure wedge through an effective-unit-weight model.
  • References: Vesic (1973); Meyerhof (1963); Hansen (1970); FHWA GEC-6 (FHWA-SA-02-054); AASHTO LRFD.

Applicability limits & conventions
  • The high-level compute() applies Vesic depth factors and averages the effective unit weight over the wedge; this runs higher than an example that sets the depth factor to 1.0 (cohesive overburden) or uses an AASHTO Cw groundwater correction — both are defensible conventions (validation V-021).

Problems it solves

4.2.1 axial_pile — Axial pile capacity

Problems it solves

Axial capacity of a driven pile in compression (and uplift): the skin friction and toe resistance developed as the pile passes through a layered profile, and how capacity grows with embedment depth.

Methods & theories implemented
  • Nordlund (1963/1979) method for shaft and toe in cohesionless soils (K–δ and correction-factor charts, bearing-capacity-factor toe with the Meyerhof qL limit).
  • Tomlinson / α-method total-stress skin friction in cohesive soils; 9·cu end bearing.
  • β-method effective-stress skin friction for all layers.
  • Groundwater-split effective-stress integration; optional per-layer separate shaft/toe friction angle; below-grade pile head (head_depth); uplift capacity.
  • Pile sections: closed/open pipe, square/circular concrete, and H-piles (with the box perimeter/toe-area convention).
  • References: FHWA GEC-12 (FHWA-NHI-16-064); Nordlund; Tomlinson; validated against the GEC-12 Nordlund/α worked examples (validation V-001/V-002).

4.4 Slope stability & continuum FEM

Will the slope stand? Limit-equilibrium factors of safety and a full 2D finite-element stress/deformation and strength-reduction engine.

4.4.1 slope_stability — Slope stability

Problems it solves

Will the slope stand? The factor of safety of an earth or rock slope against sliding — for a specified surface or the critical one found by search — under drained or undrained conditions, groundwater, surcharge, seismicity, reinforcement, and rapid drawdown, and its probabilistic FOS.

Methods & theories implemented
  • Methods of slices: Ordinary/Fellenius, Bishop simplified (1955), Janbu (corrected), Spencer (1967), Morgenstern–Price, and the rigorous GLE (Fredlund–Krahn) with interslice-force functions.
  • Critical-surface search: circular centre-grid, entry–exit arcs, random noncircular polylines with differential-evolution / PSO refinement, and weak-layer search.
  • Advanced per-layer strength models (the layer strength_model selector): Mohr–Coulomb (default), SHANSEP undrained-strength-ratio (su = S·OCRm·σ′v, with a su,min floor), and Generalized Hoek–Brown for rock (σci, GSI, mi, D).
  • Reinforcement crossing the slip surface: soil nails (pullout + bar-tension capacity, FHWA GEC-7), tieback anchors (allowable tension when the bond zone crosses the surface), horizontal geosynthetic layers, and Ito & Matsui (1975) stabilizing piles; plus ponded water, tension cracks, and pseudo-static kh.
  • Seismic displacement (Newmark sliding block + Jibson 2007 regression), rapid drawdown (USACE 2-stage / Duncan–Wright–Wong 3-stage), infinite-slope FOS, and probabilistic FOS (FOSM and Monte Carlo).
  • References: Fredlund & Krahn (1977); Bishop (1955); Spencer (1967); Janbu; Duncan, Wright & Brandon (2014); Ito & Matsui (1975); Jibson (2007); validated vs Fredlund–Krahn / ACADS / Duncan / Slide2 (VALIDATION.md, V-015/V-026…V-040).

5.2 Find with text, read with pixels

Retrieval is lexical: reference_search runs a full-text (SQLite FTS5) query over the chapter text and equation captions, expanded with a curated synonym map so a query for “passive resistance” also finds “Kp”. That roughly quadrupled recall@5 in the retrieval eval with no disturbance to the top hit. Figures are handled the same way for search, then differently for reading: figure_search finds a chart by meaning, and read_reference_figure renders the actual page and hands it to a vision model to read a value off the axes — because a design chart’s value lives in its pixels, not its caption.

Why not image embeddings?

CLIP-style image embeddings were evaluated and rejected — they are a poor fit for axis-labelled line charts. The architecture is deliberately “find lexically, read visually.” This keeps retrieval cheap and explainable while still getting an accurate value off the chart.

In agent workflows all of this is reached through the consult_references sub-agent described in Chapter 3, so the primary agent stays focused on the analysis while a scoped consultant handles citations and figure read-offs.

Chapter 6Geometry import & model setup

Real analyses start from a real cross-section — a CAD file or a PDF drawing, not a set of hand-typed coordinates. Three modules get geometry in and a fourth gets a calc package out; a fifth orchestrates a staged, human-gated model build.

6.1 DXF import

dxf_import reads a DXF, discovers its layers, and converts polylines into the SlopeGeometry and FEM inputs the analysis modules consume. The workflow is discover → parse → build: list the layers, map them to surface / soil boundaries / water table / reinforcement, then build the model. It needs the [dxf] extra (ezdxf).

Needs the [pdf] extra.

6.3 DXF export

dxf_export writes a cross-section back out to DXF — surface, soil boundaries, water table, nails/anchors, and annotations — so a model built or checked in the toolkit can return to CAD.

6.4 Staged, human-gated model setup (geo_project)

geo_project is a canonical project document for building 2D LE/FEM models in stages with confirmation gates. The agent never trusts its own read of a drawing: it ingests geometry with provenance quarantine, then renders an echo-back cross-section so a human visually confirms the numbers before they enter the model. Chapter 7Reliability & probabilistic analysis

This is the payoff of the whole architecture. Soil properties carry coefficients of variation of 10–40% — far larger than structural materials — so the honest deliverable is not a lone factor of safety but a reliability index β and a probability of failure Pf. The reliability module turns any deterministic method into a probabilistic one.

7.1 Declare the uncertainty, pick an engine

Declare the uncertain inputs as random variables, hand the engine a performance function g(values) that returns a factor of safety or a margin, and pick an engine. All four share the same g() contract:

EngineCostWhat it does
FOSMcheapFirst-Order Second-Moment — propagates means and variances for an instant β, no sampling
PEMrobustRosenblueth point-estimate — evaluates g() at ± points; handles non-linearity without a derivative
Monte Carloexact-ishSamples the joint distribution N times and counts failures; correlations via Cholesky, optional Latin Hypercube
FORMefficientNative first-order reliability — finds the design point; β at a fraction of Monte Carlo’s calls

7.2 Don’t assume a COV — look it up

cov_guidance is a queryable knowledge base of published coefficients of variation (Duncan 2000, TC304, Phoon & Kulhawy) keyed by soil property, so the input uncertainty is defensible rather than assumed. variance_reduction applies Vanmarcke spatial averaging — a footing averages over its footprint, a long slope samples many weak and strong zones, so the variance that actually matters is reduced over the volume the structure loads. combined_cov composes several sources of uncertainty.

salib wraps SALib for variance-based global sensitivity: Sobol indices and Morris screening answer which input drives the spread of the answer, so the site investigation and the design effort go where they matter. pystra adds a dedicated structural reliability engine (FORM / SORM / Monte Carlo) for limit-state functions.

8.2 How to read the verdicts

PASS

The module reproduces the published answer within the stated tolerance. Example: the Mononobe–Okabe seismic coefficient KAE reproduces the GEC-11 value to within 0.06%; the Fellenius/Bishop factors of safety match the Caltrans slice table to the printed digit.

CONVENTION

The module and the example use a defensibly different convention, and the delta is documented — the module is not tuned to the one example. Example: for the GEC-6 spread footing, the packaged high-level call applies Vesic depth factors while the example deliberately sets them to 1.0 for cohesive overburden; assembled the example’s way, the module recovers the published closed form exactly.

N/A (scope)

The module implements a different method than the example — a documented coverage gap, not a failure. Example: a slope problem whose pore pressure is supplied as an interpolated flow-net grid, which the module (piezometric surface or per-layer ru) does not accept.

The philosophy is visible in these notes: where the module and a source differ, the manual says why, in the source’s own terms, rather than quietly fitting the code to one benchmark. Several checks even surfaced errors in the published examples (for instance an under-iterated Bishop value that the module iterates to convergence).

Chapter 9Limitations & responsible use

Everything in the disclaimer at the front of this manual applies. This chapter makes the boundaries concrete, because knowing where a tool stops is part of using it well.

9.1 It is a multiplier for judgment, not a replacement

The toolkit runs the methods; it does not know your site and cannot exercise engineering judgment. A qualified, licensed professional engineer familiar with the site must independently review every input, assumption, method selection, and result before it is relied upon. Using this software creates no engineer-of-record relationship.

9.2 Validation covers only the documented cases

Agreement on a benchmark (Chapter 8) does not certify correctness for any other geometry, parameter range, or loading condition. The absence of a validation entry means a result has not been independently checked here. Read the module’s DESIGN.md and VALIDATION.md for its exact scope, sign conventions, and edge cases before trusting it on a new problem.

9.3 Known method boundaries

A few honest scope notes drawn straight from the validation record:

  • Diesel-hammer drivability in wave_equation is an energy–velocity approximation, not a GRLWEAP combustion + ram-cycle model; treat diesel bearing graphs as a trend, not a tie-out.
  • slope_stability accepts a piezometric surface or per-layer ru, not an arbitrary interpolated pore-pressure grid; its tension crack is entry-side and keeps the cracked wedge as driving soil (a conservative convention).
  • ground_improvement uses an equal-strain settlement-reduction model for aggregate piers, not the GEC-13 stiffness-modulus two-layer method; the Priebe factor depends on the area-ratio and column-friction conventions you feed it.
  • The high-level bearing_capacity call applies Vesic depth factors and an effective-unit-weight groundwater model — defensible, but different from an AASHTO Cw groundwater correction; know which convention your check needs.
  • Soldier-pile arching and the FHWA apparent-diagram single-anchor embedment solver are documented gaps in soe/sheet_pile.

9.4 Treat agent output with particular care

The LLM agent can select the wrong method, mis-transcribe a value, or read a chart imperfectly. Its answers are a starting point for review, never a final basis for design. Inspect the tool calls (result.tool_calls, or verbose=True) — the methods and parameters actually used — not just the prose. Confirm the inputs and units: everything is SI, and mis-scaled units are the user’s responsibility to catch.
