GeotechStaffEngineer · A developer's overview

Soil is uncertain.
So you don't calculate once — you calculate thousands of times.

Geotechnical engineering is the science of building on and out of the ground — foundations, walls, slopes, excavations. The ground is variable and only partly known, so a single number is never the answer; engineers must sweep the parameter space to understand a problem's depth and variability. This project packages decades of those methods as composable, machine-callable Python — then hands them to an LLM agent. It's the next turn of a very old wheel: using computers to do more engineering, faster, and to explore problems we couldn't afford to before.

The problem domain

01What a geotechnical engineer actually does

Before the code makes sense, the job has to. Geotechnical engineers decide whether the ground can carry what we want to put on it — and design the foundation, wall, or slope that makes it safe. Unlike a steel beam with a known, certified strength, their material is the earth itself: heterogeneous, layered, partly saturated, and sampled at only a handful of points across an entire site.

1 · Characterize

Drill borings, push CPT cones, run lab tests. Out comes a sparse, noisy picture of strength, stiffness, and groundwater — never the whole truth.

2 · Idealize

Collapse that data into a layered soil profile: design values of friction angle φ, cohesion c, unit weight γ, water table — each an estimate with a spread.

3 · Analyze

Run the method — bearing capacity, settlement, pile capacity, slope stability — usually a chain of code-prescribed formulas from DM7, FHWA, UFC, AASHTO.

4 · Check & revise

Compare against the required factor of safety. Vary the assumptions. Did groundwater rise? Is φ 2° lower? Re-run. Loop until the design is robust across the uncertainty.

The crux — steps 3 and 4 are not one calculation; they are the same calculation run again and again across plausible soil states. The engineering judgment is in choosing the range and reading the spread of answers — not in any single result.

Depth & variability

02Why the answer is a distribution, not a number

Two engineers, same site, can pick φ = 30° or φ = 33° and both be defensible — the soil genuinely varies that much, spatially and from test scatter. A bearing capacity that doubles between those two values is not a rounding question; it's the whole question. Understanding a geotechnical problem means understanding how the answer moves as the inputs move.

Parameter uncertainty

Each design property is an estimate with a coefficient of variation (COV). Strengths commonly carry 10–40% COV — far larger than structural materials.

Spatial variability

Soil changes from boring to boring. A footing averages properties over its footprint; a long slope samples many weak and strong zones. Scale matters.

Model & scenario sweeps

Which method? Drained or undrained? Water table high or low? Seismic case? Each is another axis to run the calculation along.

Done by hand, that exploration is slow and expensive, so it gets truncated — a couple of cases, a fixed factor of safety, and a lot of conservatism to cover the gaps. Done by machine, the same deterministic method can run ten thousand times across the full statistical picture, returning a reliability index β and a probability of failure Pf instead of a lone FOS. That shift — from one point estimate to the shape of the whole answer — is the emphasis this project is built around.

The intent

03Computers have always amplified the engineer

Every tool in this lineage did the same thing: it let one engineer explore more of the problem in the time they had. This project is the next step in that line — not a replacement for judgment, but a force multiplier for it.

The thesis in one line — take the methods a staff engineer uses, make every one of them a clean Python function that returns structured data, wrap the variability tooling around them, and put a reasoning agent on top. The point is not to remove the engineer — it's to let them ask bigger questions and check more of the problem than a deadline ever allowed.

How it's built

04One stack: a shared soil spine, deterministic methods, a variability layer, references, an agent

The codebase is layered so each concern stays independent. Analysis modules never import each other — they all speak one shared SoilProfile. The reliability and sensitivity engines wrap any deterministic method as a callable. The reference library and the agent harness sit on top. Read it bottom-up.

Why a shared spine

One SoilProfile type, with adapters into each module's inputs, means modules compose without knowing about each other — and an agent only has to learn one way to describe the ground.

Why no cross-imports

Analysis modules are independent leaves. That keeps each testable in isolation, swappable, and safe for an agent to call à la carte — the routing that combines them lives in the agent layer, not inside a module.

Why variability is its own layer

Because it's orthogonal. reliability doesn't know what bearing capacity is — it knows how to take any callable g(values) and characterize how its output scatters. Wrap once, apply everywhere.

The emphasis

05The variability engine — the same calculation, thousands of times

This is the heart of the “explore the problem, don't point-estimate it” idea. The reliability module turns any deterministic analysis into a probabilistic one. You declare the uncertain inputs as random variables, hand it a performance function g(values) → FOS or margin, and pick an engine. It returns the reliability index β and probability of failure Pf — the distribution of the answer, not a single value.

from reliability import RandomVariable, monte_carlo, fosm, cov_guidance
from reliability.wrappers import bearing_capacity_reliability

# 1 · Uncertain inputs — pull a defensible COV straight from the literature
cov_guidance("friction_angle")        # → Duncan 2000 / Phoon-Kulhawy guidance, ~0.05–0.15
phi   = RandomVariable("phi",   mean=32, cov=0.10, dist="lognormal")
c     = RandomVariable("c",     mean=8,  cov=0.30, dist="lognormal")
gamma = RandomVariable("gamma", mean=18, cov=0.07)

# 2 · The performance function: the DETERMINISTIC module, called per sample
def g(v):                                  # v = {"phi":.., "c":.., "gamma":..}
    res = bearing_capacity_analysis(phi=v["phi"], c=v["c"], gamma=v["gamma"], width=2.0)
    return res["factor_of_safety"]            # scalar margin: g ≥ 1 is "safe"

# 3 · Pick an engine — all four share the same g() contract
mc   = monte_carlo(g, [phi, c, gamma], n=50_000, target=1.0)
fast = fosm(g, [phi, c, gamma])           # first-order, ~2·n_vars calls, no sampling

print(mc.beta, mc.pf)     # reliability index & probability of failure — the real deliverable

FOSM cheap

First-Order Second-Moment. A few calls per variable; propagates means & variances. Instant β when sampling is overkill.

PEM robust

Rosenblueth Point Estimate — evaluates g() at ± points. Handles moderate non-linearity without a derivative.

Monte Carlo exact-ish

Sample the joint distribution N times, count failures. The ground truth the others approximate — correlations and all.

FORM efficient

Native first-order reliability — finds the design point on the limit-state surface. β at a fraction of MC's calls.

Priors, not guesses — cov_database

So an engineer (or an agent) doesn't invent a COV, the module ships a queryable knowledge base of published coefficients of variation (Duncan 2000, TC304, Phoon & Kulhawy) keyed by soil property. Ask, don't assume.

Scale-aware — spatial

Vanmarcke spatial averaging reduces variance over the volume a structure actually loads: a wide footing “sees” averaged soil, a borehole sees point scatter. scale_of_fluctuation_guidance + variance_reduction make that explicit.

Complementing reliability, salib_agent answers the other half of “how does the answer move” — global sensitivity (Sobol, Morris): which input drives the variance? Together they turn one calculation into a full map of a problem's depth and the factors that govern it.

Why you can trust the numbers

07Engineering rigor, enforced by convention and tests

An agent that confidently returns a wrong bearing capacity is worse than useless. The project's discipline is what makes the outputs citable: consistent units, structured I/O, and validation against the same published benchmarks an engineer would defend their work with.

One unit system, one I/O shape

Everything is SI (m, kPa, kN, degrees). Every analyze_*() returns a dataclass with .summary() → str (human/agent reading) and .to_dict() → dict (machine/JSON). Predictable in, predictable out.

Validated against the literature

Flagship modules carry a VALIDATION.md with worked checks: slope_stability vs Fredlund & Krahn 1977 / ACADS / Duncan; fem2d vs Griffiths & Lane and the Prandtl solution (~2%); lateral pile vs a COM624P oracle suite.

Tests as a safety net

Each module ships a pytest suite sized to its risk — e.g. slope_stability 384, fem2d 353, geotech_common 288, and the reference library alone 3,529. Changes regress against textbook answers, not vibes.

A reviewer in the loop

The agent can run a second, reference-scoped pass that checks methodology, parameter ranges, and required safety factors against the standards — the digital echo of a senior engineer's red pen.

Using it

08The developer surface — three ways in

The same methods are reachable at three altitudes: call a module directly for a deterministic result, wrap it in the variability engine for β/Pf, or let the agent choose and chain methods from a natural-language ask. Pick the altitude that fits the task.

A · deterministic — import a module, get a dataclass back
from bearing_capacity import Footing, SoilLayer, BearingSoilProfile, BearingCapacityAnalysis

footing = Footing(width=2.0, length=10.0, depth=1.5, shape="strip")
layer   = SoilLayer(friction_angle=30, cohesion=10, unit_weight=18, thickness=10)
profile = BearingSoilProfile(layer1=layer, gwt_depth=5.0)

result = BearingCapacityAnalysis(footing=footing, soil=profile).compute()
print(result.summary())     # readable report
result.to_dict()            # JSON-ready for an agent or a calc package
B · probabilistic — same calc, wrapped to return the distribution (see §05)
from reliability.wrappers import bearing_capacity_reliability
out = bearing_capacity_reliability(variables=spec, engine="monte_carlo", n=50_000)
out.beta, out.pf            # the answer's shape, not one point on it
C · agentic — describe the problem, let the agent run the methods & cite the standards
from funhouse_agent import GeotechAgent, NativeToolEngine
agent = GeotechAgent(genai_engine=NativeToolEngine(fh_prompter))     # Claude / Funhouse Prompter / OpenAI-native

agent.ask("2 m strip footing, 1.5 m deep, sand phi=30, c=10, water at 5 m. "
          "Bearing capacity and FOS — and how sensitive is it to phi? Cite the method.")
# → picks bearing_capacity, runs it, sweeps phi, consults DM7/GEC, returns a cited answer
The throughline — at every altitude it's the same validated method underneath. The deterministic call is the atom; the variability engine runs that atom across uncertainty; the agent decides which atoms to run and reads the standards back to you. More of the problem, explored faster — which is all a computer has ever been for.
