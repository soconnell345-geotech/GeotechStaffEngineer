# 6.0.0 — Single-namespace restructure (`geotech_staff_engineer.*`) — effort assessment + execution plan

## Context

Today the wheel installs ~36 top-level packages (`bearing_capacity`, `settlement`,
`reliability`, `webapp`, …) — squatting generic module names that can silently
collide with other PyPI packages in a shared environment (a real `reliability`
package exists). The fix is the standard layout: one `geotech_staff_engineer/`
package with everything under it, plus compatibility stubs so nothing external
breaks on day one. Assessed and parked at 5.8.1; the owner now wants the effort
sized (and this plan is written to be executable by the next agent).

## Effort verdict

**One focused agent session: roughly 4–8 hours of agent time, moderate-heavy
token spend** (the rewrite itself is scripted and fast; the cost is 2–3 full
gate runs at ~25–40 min each plus shim/edge iteration). It is NOT risky in the
"unknown unknowns" sense — every sharp edge is now enumerated above — but it is
absolutely a dedicated-session, full-gate, owner-releases-it job, not a side
task. Do it at a quiet moment (not mid-deployment-debugging), version it 6.0.0.

## Execution plan (for the next agent)

1. **Branch** (`v6.0-restructure`), work in the worktree per repo convention.
2. **Move**: `git mv` the 36 packages (+ `groundhog_agent.py`) under
   `geotech_staff_engineer/`; add its `__init__.py` (version + lazy nothing —
   keep import cheap).
3. **Scripted rewrite** (write ONE Python script, keep it in module_work/ for
   audit): regex `^(\s*)(from|import)\s+(<name>)\b` → prefixed form across all
   .py under geotech_staff_engineer/; then the string surfaces: MODULE_REGISTRY
   adapter values, `_import_map`, `files("funhouse_agent")` →
   `files("geotech_staff_engineer.funhouse_agent")`, entry point, test mirrors.
4. **Hand-fix the path traps** (worked_examples resolve_source_pdf: compute
   repo root as `parents[2]` of `__file__` AND keep the env override primary;
   webapp/app.py bootstrap: one more `dirname` + prefixed imports; the 5 tests).
5. **pyproject rewrite**: `include = ["geotech_staff_engineer*"]`, package-data
   keys re-prefixed, testpaths → `geotech_staff_engineer/...` (keep
   `geotech-references/tests` as-is), entry point, drop py-modules (module file
   moves inside).
6. **Compatibility stubs** (the zero-day-breakage guarantee): generate 36 tiny
   top-level stub packages, each `<name>/__init__.py`:
   `import geotech_staff_engineer.<name> as _m, sys; sys.modules[__name__] = _m`
   preceded by a `DeprecationWarning`. Ship them in the wheel (tiny); the
   owner's Foundry `app.py` (`from webapp.foundry_entry import main`),
   Databricks notebooks, and any user scripts keep working untouched. Remove in
   7.0.
7. **Docs**: FOUNDRY.md snippet → new import (note old form still works via
   stub), Start Geotech App.bat, webapp/README, README examples,
   funhouse_agent_guide (18 snippets), CLAUDE.md architecture section.
8. **Verify**: full gate (all chunks — expect ~10k tests green, zero behavior
   change); wheel-content check (single top-level package + stubs, package-data
   present, no PDFs); fresh-venv install of the built wheel + smoke: old-style
   `from webapp.foundry_entry import main` (stub path, warns), new-style
   imports, `call_agent` dispatch, `geotech-disclaimer` script, worked-example
   page retrieval with `GEOTECH_REFERENCES_DOCS` set and unset.
9. **Release 6.0.0 on the owner's word**; owner's Foundry upgrade is the usual
   maestro bump + Publish and sync (their app.py needs NO edit thanks to
   stubs — update it to the new import at leisure).

## Rollback / safety

Everything is one branch + one release; the stubs make the wheel
backward-compatible, so a bad discovery post-release is fixable forward
(6.0.1) rather than requiring a yank. The submodule is untouched. Standing
rules apply: no tag without the owner's word; full gate before it.
